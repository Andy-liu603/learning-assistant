# 后端技术选型说明

> 学习助手 v2.4 — 技术决策记录

## 概述

本文档记录学习助手后端的技术选型理由。每个决策都从项目背景（个人使用、本地部署、快速迭代）出发，说明"为什么选这个"和"为什么不选那个"。

---

## 1. Web 框架：Flask vs FastAPI vs Django

### 选择：Flask 3.1

| 维度 | Flask | FastAPI | Django |
|------|-------|---------|--------|
| 学习曲线 | 低 | 中 | 高 |
| 单用户场景适配 | 优秀 | 过度设计 | 过重 |
| 生态丰富度 | 高（蓝图、扩展） | 中 | 高（但很多用不上） |
| 异步支持 | 有限（需额外配置） | 原生 | 需配置 |
| 部署复杂度 | 简单（一行启动） | 需 ASGI 服务器 | 复杂 |

**选择理由**：
- **个人项目不需要异步**：FastAPI 的核心优势是异步 I/O，但对于单用户本地部署的场景，Flask 的同步模型完全足够
- **轻量但功能完整**：Flask 的 Blueprint 机制可以良好组织代码，`flask-cors`、`flasgger` 等扩展即插即用
- **快速原型验证**：Flask 的"最小可行"哲学与项目 MVP 阶段的需求一致

**为何不选**：
- FastAPI：异步对单用户场景无收益，自动生成的 OpenAPI 文档虽好，但 Flasgger 也能做到
- Django：ORM、Admin、模板引擎等"电池"在本项目中都用不上，徒增复杂度

---

## 2. 数据库：SQLite vs PostgreSQL vs MySQL

### 选择：SQLite

**选择理由**：
- **零配置部署**：不需要安装数据库服务器，项目文件夹即数据库
- **单用户足够**：SQLite 在并发写入 1-2 的场景下性能不输 PostgreSQL
- **WAL 模式**：启用 `PRAGMA journal_mode=WAL` 后读写可并发，读不阻塞写
- **备份方便**：整个数据库就是一个 `.db` 文件，复制即备份

**为何不选**：
- PostgreSQL：功能强大但需要额外安装和配置，对本项目的规模来说是"用大炮打蚊子"
- MySQL：与 PostgreSQL 类似，且 SQLite 在嵌入式场景下更成熟

**注意事项**：
- SQLite 在并发写入 > 10 时会显著下降（当前项目写入量极低，不构成瓶颈）
- 如果未来迁移到多用户场景，SQLite → PostgreSQL 的数据迁移也已规划在 v3.0 路线图中

## 3. 向量数据库：ChromaDB vs Pinecone vs Milvus

### 选择：ChromaDB 1.2.2

| 维度 | ChromaDB | Pinecone | Milvus |
|------|----------|----------|--------|
| 部署 | 本地，零配置 | 云服务 | 需 Docker |
| 成本 | 免费 | 按量付费 | 免费（需服务器资源） |
| 嵌入模型集成 | 内置 Sentence-Transformers | 需外部集成 | 需外部集成 |
| 查询速度 | 快（HNSW 索引） | 快 | 快 |
| 数据量上限 | 中等（百万级） | 无限制 | 无限制 |

**选择理由**：
- **本地优先**：整个项目设计为完全离线可用，ChromaDB 支持本地持久化
- **开发体验**：pip install 即可使用，无需启动额外服务
- **嵌入模型集成**：ChromaDB 直接支持 Sentence-Transformers，减少胶水代码
- **Collection 隔离**：按 `user_{uid}_doc_{did}` 命名 collection，天然支持多用户数据隔离

**为何不选**：
- Pinecone：需要网络连接和付费，违背"本地运行"的定位
- Milvus：功能强大但需要 Docker 部署，增加运维负担

**HNSW 参数调优**：
- `hnsw:space`: cosine（与 Sentence-Transformer 余弦输出匹配）
- `hnsw:M`: 16（默认值，在精度和构建时间间平衡）
- `hnsw:ef_construction`: 100（默认值，对个人文档规模足够）

## 4. LLM 集成：多 Provider 统一架构

### 选择：OpenAI 兼容协议 + 多 Provider 热切换

**架构设计**：
```python
# 每个 Provider 统一使用 openai 库的 client
client = OpenAI(api_key=api_key, base_url=base_url)

# 通过 MODEL_PROVIDERS 配置实现动态注册
providers = [
    {"name": "DeepSeek", "type": "text", "base_url": "...", "model": "deepseek-chat"},
    {"name": "智谱 GLM-4.6V", "type": "multimodal", "base_url": "...", "model": "glm-4.6v"}
]
```

**选择理由**：
- **避免供应商锁定**：任何支持 OpenAI 兼容接口的模型都能接入
- **热切换**：模型切换不需要重启服务，`set_global_model()` 即时生效
- **多模态自动路由**：`chat_with_image()` 自动切换到 `type=multimodal` 的 provider
- **网络容错**：自动重试 3 次（2s/4s/6s 间隔），连接失败时优雅降级

**为何不直接调用各模型原生 API**：
- 每个模型 API 签名不同（如 DeepSeek 和 GLM 的参数名不一致）
- 维护多套适配代码成本高
- OpenAI 兼容协议已成为事实标准

## 5. 嵌入模型：本地 Sentence-Transformer

### 选择：bge-small-zh-v1.5（首选），all-MiniLM-L6-v2（回退）

**选择理由**：
- **完全离线**：模型下载到本地后设置 `HF_HUB_OFFLINE=1`，无需联网
- **中文优化**：BGE 系列是中文嵌入的 SOTA，small 版本在速度和效果间平衡
- **ONNX 加速**：Sentence-Transformer 优先使用 ONNX 后端，推理速度提升约 2x
- **向量维度适中**：BGE 512 维 / MiniLM 384 维，在语义召回和存储效率间平衡

**模型对比**：

| 指标 | BGE-small-zh-v1.5 | all-MiniLM-L6-v2 |
|------|-------------------|-------------------|
| 向量维度 | 512 | 384 |
| 中文效果 | 优秀（专为中文优化） | 良好（通用多语言） |
| 模型大小 | ~100MB | ~90MB |
| 推理速度 | 中 | 快 |

## 6. 异步处理：ThreadPoolExecutor

### 选择：ThreadPoolExecutor（而非 Celery）

**选择理由**：
- 异步场景少：文档解析（2 线程）、资讯抓取（2 线程）
- 不需要消息队列：任务量小，不需要 Celery + Redis 的重型方案
- 简单可靠：标准库自带，无额外依赖

**使用场景**：
- 文档上传后异步解析 + 向量化（避免阻塞 HTTP 响应）
- 资讯批量抓取（并发请求多个 RSS 源）
- 测评出题 + 知识点提取并发执行

---

## 7. 认证：JWT (HS256)

### 选择：PyJWT + werkzeug.security

**选择理由**：
- 无状态认证，无需服务端 session
- werkzeug 的 `pbkdf2:sha256` 密码哈希安全可靠
- 7 天过期时间 + 刷新机制

**安全加固（v2.4）**：
- 去掉 JWT_SECRET 硬编码默认值 → 启动时自动生成随机密钥（若无配置）
- 添加 Flask-Limiter 速率限制 → 防止暴力破解

## 总结

后端技术选型的核心原则是 **"刚好够用"**。每个选择都从项目实际需求出发，不追求"最佳实践"而是追求"最合适"。这本身也是产品经理的核心能力：在约束条件下做选择，用最小成本验证最大价值。
