# 前端技术选型说明

> 学习助手 v2.4 — 前端技术决策记录

## 概述

学习助手的前端采用**原生 JavaScript SPA**，不依赖任何前端框架（React/Vue/Angular）。这个选择在 2026 年的前端生态中看起来"反主流"，但它是一个经过深思熟虑的产品决策——正如产品经理在 MVP 阶段选择"不做"比选择"做什么"更需要勇气。

---

## 1. UI 框架：Vanilla JS vs React vs Vue

### 选择：原生 JavaScript SPA

| 维度 | Vanilla JS | React | Vue |
|------|-----------|-------|-----|
| 学习成本 | 低（JS 基础即可） | 中-高（JSX/Hooks/状态管理） | 中（模板语法/响应式） |
| 构建工具 | 不需要 | Vite/Webpack | Vite/Webpack |
| 打包大小 | 0 KB | ~40 KB（React + ReactDOM） | ~30 KB |
| 开发速度（个人） | 快（所见即所得） | 中（需要构建步骤） | 中 |
| 生态丰富度 | 低 | 极高 | 高 |
| MVP 阶段适配 | 优秀 | 一般 | 一般 |
| 团队协作适配 | 差 | 优秀 | 优秀 |

**选择理由——从产品经理视角**：

#### A. 这不是"不会 React"的选择

项目主人（北航硕士、准备 AI PM 求职）有 React + TypeScript + Vite + Tailwind CSS 的技术栈背景。选择 Vanilla JS 是**有意识的决策**，不是技术能力的限制。

#### B. MVP 阶段的核心逻辑

MVP 阶段的目标是**快速验证核心假设**（"AI + RAG 能提升学习效率吗？"），而非构建一个可扩展的前端架构。

- React 的学习成本（虚拟 DOM、Hooks 闭包陷阱、状态管理）在单页应用中没有收益
- Vite 的 HMR 在个人开发场景中并非必需
- 零构建工具意味着零构建配置时间，修改 CSS/JS 后刷新即生效

#### C. 模块化不是框架的专利

```javascript
// 每个页面都是独立的 ES Module，按需动态导入
const pages = {
  dashboard: () => import('./pages/dashboard.js').then(m => m.renderDashboardPage()),
  chat:      () => import('./pages/chat.js').then(m => m.renderChatPage()),
  // ...8 个页面模块
};
```

ES Modules + 动态导入已经实现了懒加载、代码分割等"框架特性"。

#### D. 什么时候会用 React？

当项目需要以下特性时，迁移到 React 就是合理的：
- 多个开发者协作（组件化约束、类型检查）
- 复杂的交互状态（表单验证、实时协作）
- 需要 SSR/SSG（SEO、首屏性能）

这些目前在**个人学习助手**的场景中都不需要。v3.0 如果做多人协作，会考虑 React 迁移。

## 2. SPA 路由：Hash-based

### 选择：`window.location.hash` + `hashchange` 事件

**选择理由**：
- 无需服务端路由配置（所有路由由前端 JS 处理）
- URL 中的 `#` 不会触发页面刷新
- 兼容性好，所有浏览器都支持
- 对于本地 Flask 静态服务，Hash 路由是最简单的 SPA 方案

**路由表**：
```
#/dashboard   -> 学习仪表盘（默认首页）
#/chat        -> 智能对话
#/plan        -> 学习计划
#/news        -> AI 资讯追踪
#/library     -> 资料库
#/quiz        -> 知识测评
#/progress    -> 学习进度
#/report      -> 学习报告
#/settings    -> 系统设置
#/login | #/register -> 认证页面（公开）
```

## 3. 图表方案：ECharts 5.5

### 选择：ECharts（而非 Chart.js 或 D3.js）

**选择理由**：
- **中文文档完善**：ECharts 是百度开源的国产图表库，中文文档质量高
- **开箱即用**：环形图、热力图、折线图一行代码初始化
- **CDN 加载**：通过 jsDelivr CDN 引入，无需打包
- **响应式**：内置 resize 机制，移动端自适应

**使用场景**：
| 图表类型 | 用途 | 页面 |
|---------|------|------|
| 日历热力图 | 90 天学习活跃度 | 仪表盘 |
| 折线图 | 8 周学习趋势 | 仪表盘 |
| 环形图（Donut） | 五级掌握度分布 | 仪表盘、进度 |

## 4. 状态管理：无框架方案

### 选择：localStorage + 闭包变量 + IndexedDB

| 存储 | 用途 | 生命周期 |
|------|------|---------|
| localStorage | JWT token、用户信息 | 持久化 |
| IndexedDB | 聊天记录（`ai_learning_chat`） | 持久化 |
| 闭包变量 | 页面内临时状态（convId、currentFilter） | 会话 |
| sessionStorage | 跨页面参数传递 | 会话 |
| `window.*` | 全局事件回调 | 页面生命周期 |

**选择理由**：
- 应用状态简单：用户信息 + 聊天记录，不需要 Redux/MobX/Zustand
- 跨页面通信场景少：仅测评页面偶尔需要从资讯页面传递参数
- localStorage 够用：认证状态管理不需要框架

## 5. CSS 架构：无框架设计系统

### 选择：CSS 变量 + 模块化 CSS 文件（4 个文件）

**文件组织**：
```
css/
├── reset.css       # 设计令牌（CSS变量） + 全局重置 + 工具类
├── layout.css      # 应用布局 + 顶部导航 + 认证页面
├── components.css  # 可复用组件（按钮/表单/卡片/徽章/弹窗）
└── pages.css       # 各页面专属样式 + 响应式
```

**设计令牌（CSS Variables）**：
```css
:root {
  --ivory:  #FBF9F5;    /* 米白底 */
  --camel:  #C4A97D;    /* 驼色 */
  --rust:   #D4652A;    /* 锈红（主色调） */
  --forest: #4A6B53;    /* 森林绿 */
  --ink:    #2C2C2A;    /* 墨黑 */
  --font-display: 'Bodoni Moda', serif;  /* 展示字体 */
  --font-body: 'PingFang SC', sans-serif; /* 正文字体 */
}
```

**选择理由**：
- 不使用 Tailwind CSS 是因为团队规模=1，不需要工具类防命名冲突
- 不使用 CSS-in-JS 是因为没有 JS 框架
- CSS 变量的可维护性对个人项目足够

---

## 6. 构建工具：不需要

### 选择：零构建工具

**选择理由**：
- ES Modules 原生支持模块化，浏览器直接执行
- 文件数量少（18 个文件），冷启动时间可忽略
- 修改后 `<script>` 标签中的版本号 `?v=N` 用于缓存破坏

**何时需要构建工具**：
- 引入 npm 生态的库（如要用 React，必须先装构建工具）
- 生产部署需要压缩/优化
- 需要 TypeScript 类型安全

## 7. 模块组织

```
js/
├── app.js          # 主路由控制器（Hash SPA）
├── api.js          # 统一 API 封装（fetch + SSE 流式）
├── auth.js         # 认证模块（登录/注册 + Token 管理）
├── utils.js        # 工具函数（日期/Toast/Markdown 渲染）
├── components.js   # 可复用组件（顶栏/确认弹窗/空状态）
├── chat-db.js      # IndexedDB 聊天持久化
└── pages/          # 8 个页面模块
    ├── dashboard.js
    ├── chat.js
    ├── plan.js
    ├── news.js
    ├── library.js
    ├── quiz.js
    ├── progress.js
    ├── report.js
    └── settings.js
```

每个页面模块导出一个 `render*Page()` 函数，由 `app.js` 统一调度。

---

## 总结

前端技术选型体现了一个产品决策原则：**在约束条件下做最合适的选择，而非追求技术潮流**。Vanilla JS + Hash SPA 在个人 MVP 阶段是最优解，它让开发者专注于核心功能（RAG 对话、测评、进度追踪），而非框架配置和构建工具调试。

如果面试官问"为什么不用 React"，答案不是"我不会"，而是"MVP 阶段不需要"。这种判断力，恰恰是产品经理需要的。
