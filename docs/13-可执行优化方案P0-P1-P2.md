## P0 ｜ 统一 ChromaDB 向量集合

### P0.1 背景与问题

`backend/services/vector_store.py` 当前为**每个文档创建独立 collection**：

```python
# 现状
collection_name = f"user_{user_id}_doc_{doc_id}"   # 多用户
# 或
collection_name = f"doc_{doc_id}"                    # 单用户
```

跨文档检索（`search()` 不传 `doc_id`）的实现是**遍历所有集合 + 逐个 query + 应用层合并排序**：

```python
prefix = f"user_{user_id}_" if user_id else "doc_"
all_collections = [c for c in self.client.list_collections() if c.name.startswith(prefix)]
for coll in all_collections:
    result = coll.query(...)          # 每个集合都查一次
    all_results.append(result)
results = self._merge_results(all_results, top_k)   # 再合并排序
```

**痛点**：文档越多跨文档问答越慢（N 个集合 = N 次检索）；每个集合都建一份 HNSW 索引（内存/磁盘放大）；`_merge_results` 在应用层二次排序（跨集合分数不可比）。

### P0.2 设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 集合粒度 | **每用户一个集合** `user_{uid}_docs` | 兼顾"统一集合"效率收益与多用户隔离；比全局单集合更易按用户清理 |
| 单用户兜底 | `learning_docs` | 保留 `user_id=None` 本地路径 |
| 文档隔离 | collection 内 `where={"doc_id": doc_id}` 过滤 | 取代"每文档独立集合"；向量层过滤，分数可比 |
| 公开 API | **完全不变** | `add_chunks/search/delete_document/get_document_count` 签名不变 → 调用方零改动 |
| 迁移触发 | 启动自动（幂等）+ 手动脚本 | 对齐现有 `run_migrations()` 的"启动即迁移"约定 |

### P0.3 改动文件

| 操作 | 文件 | 说明 |
|------|------|------|
| 修改 | `backend/services/vector_store.py` | 集合命名统一 + `search` 单集合 `where` 过滤 + `delete_document` 改 `delete(where=)` + `get_document_count` 改 `get(where=)` + 新增 `migrate_legacy_collections()` |
| 修改 | `backend/app.py` | 启动后调用 `VectorStore().migrate_legacy_collections()`（try/except 包裹） |
| 新增 | `backend/migrations/migrate_vector_store.py` | 独立可重跑的迁移脚本 |

> 无需改动：`document_service.py`、`document_routes.py`、`news_routes.py`、`database.py`、任何前端文件。

### P0.4 实施细节

**① 集合命名与获取**（替换 `_get_or_create_collection`）

```python
def _collection_name(self, user_id: int = None) -> str:
    return f"user_{user_id}_docs" if user_id else "learning_docs"

def _get_or_create_collection(self, user_id: int = None):
    return self.client.get_or_create_collection(
        name=self._collection_name(user_id),
        metadata={"hnsw:space": "cosine"}
    )
```

**② `add_chunks`**（签名不变，仅换集合）

```python
def add_chunks(self, doc_id: int, chunks: List[str], user_id: int = None) -> List[str]:
    if not chunks:
        return []
    collection = self._get_or_create_collection(user_id)
    embedding_fn = self._get_embedding_function()
    embeddings = embedding_fn(chunks).tolist()
    ids = [f"chunk_{doc_id}_{i}" for i in range(len(chunks))]
    metadatas = [{"doc_id": doc_id, "chunk_index": i, "user_id": user_id}
                 for i in range(len(chunks))]
    batch_size = 100
    for start in range(0, len(chunks), batch_size):
        end = start + batch_size
        collection.add(
            ids=ids[start:end],
            embeddings=embeddings[start:end],
            documents=chunks[start:end],
            metadatas=metadatas[start:end]
        )
    return ids
```

**③ `search`**（核心优化：N 次 → 1 次）

```python
def search(self, query: str, doc_id: int = None, top_k: int = 5,
           user_id: int = None) -> List[Tuple[str, float, dict]]:
    embedding_fn = self._get_embedding_function()
    query_embedding = embedding_fn([query]).tolist()
    collection = self._get_or_create_collection(user_id)
    where = {"doc_id": doc_id} if doc_id is not None else None
    try:
        results = collection.query(
            query_embeddings=query_embedding,
            n_results=top_k,
            where=where
        )
    except Exception:
        return []
    if not results or not results.get("documents") or not results["documents"][0]:
        return []
    documents = results["documents"][0]
    distances = results.get("distances", [[]])[0]
    metadatas = (results.get("metadatas", [[]])[0]
                 if results.get("metadatas") else [{}] * len(documents))
    return [
        (doc, 1.0 - dist, meta or {})
        for doc, dist, meta in zip(documents, distances, metadatas)
    ]
```
> 删除 `_merge_results()` 方法。跨文档检索现在是一条 `where=None` 的 query。

**④ `delete_document`**（集合保留，仅删该文档向量）

```python
def delete_document(self, doc_id: int, user_id: int = None):
    try:
        collection = self._get_or_create_collection(user_id)
        collection.delete(where={"doc_id": doc_id})
    except Exception:
        pass
```

**⑤ `get_document_count`**（用 `get(where=)` 计数）

```python
def get_document_count(self, doc_id: int, user_id: int = None) -> int:
    try:
        collection = self._get_or_create_collection(user_id)
        res = collection.get(where={"doc_id": doc_id}, include=[])
        return len(res.get("ids", []))
    except Exception:
        return 0
```

**⑥ 旧数据迁移方法**（幂等，加到 `vector_store.py`）

```python
import re

def migrate_legacy_collections(self):
    """一次性迁移旧版每文档独立集合 → 统一集合。幂等：无旧集合则直接跳过。"""
    legacy = re.compile(r"^(?:user_(\d+)_)?doc_(\d+)$")
    old_colls = [c for c in self.client.list_collections() if legacy.match(c.name)]
    if not old_colls:
        log.info("[VectorStore] 无旧版集合，跳过迁移")
        return
    for old in old_colls:
        m = legacy.match(old.name)
        uid = int(m.group(1)) if m.group(1) else None
        try:
            data = old.get(include=["embeddings", "documents", "metadatas"])
        except Exception as e:
            log.error(f"读取旧集合 {old.name} 失败: {e}")
            continue
        ids = data.get("ids", [])
        if not ids:
            self.client.delete_collection(old.name)
            continue
        target = self._get_or_create_collection(uid)
        existing = set(target.get(ids=ids, include=[]).get("ids", []))
        add_ids, add_emb, add_docs, add_meta = [], [], [], []
        for i, cid in enumerate(ids):
            if cid in existing:
                continue
            add_ids.append(cid)
            add_emb.append(data["embeddings"][i])
            add_docs.append(data["documents"][i])
            add_meta.append(data["metadatas"][i])
        if not add_ids:
            self.client.delete_collection(old.name)
            continue
        target.add(ids=add_ids, embeddings=add_emb,
                   documents=add_docs, metadatas=add_meta)
        if set(target.get(ids=add_ids, include=[]).get("ids", [])) == set(add_ids):
            self.client.delete_collection(old.name)
            log.info(f"已迁移并清理旧集合 {old.name}（{len(add_ids)} 条）")
        else:
            log.warning(f"迁移校验未通过，保留旧集合 {old.name}")
```

## P1 ｜ 嵌入 ONNX 加速

### P1.1 现状（关键发现）

`vector_store.py` 的 `_get_embedding_function()` **已经写了 ONNX 路径**，但 `requirements.txt` 缺 `optimum` + `onnxruntime`，导致 `try` 抛异常、一直回退 PyTorch：

```python
# 现状（缺依赖 → 永远走 except 分支）
try:
    model = SentenceTransformer(self.embedding_model, backend="onnx")
    print(f"[VectorStore] ONNX 加速已启用: {self.embedding_model}")
except Exception:
    model = SentenceTransformer(self.embedding_model)   # ← 实际一直走这
```

官方文档（sbert.net）确认：`backend="onnx"` 需要 `sentence-transformers>=3.2`（我们 4.1.0 ✓），且仅在模型目录**已含 ONNX 格式**时自动复用，否则首次加载时本地导出。模型已在本地缓存（HF_HUB_OFFLINE=1 下可用），**离线可导出，不需要联网**。

### P1.2 目标

装上 ONNX 依赖，让嵌入编码真正走 ONNX Runtime（官方基准 2–3x 提速，尤其输入长度差异大时收益更高）。

### P1.3 改动（极小）

**① `requirements.txt` 新增一行**

```text
sentence-transformers[onnx]>=4.1.0    # 自动拉取 optimum + onnxruntime
```
> 也可显式写 `optimum>=1.20` + `onnxruntime>=1.18`；`[onnx]` extra 已包含二者。

**② （可选）规范日志 + 缓存 ONNX 模型避免重复导出**

把 `print` 换成 `log.info`（符合项目 logging 约定），并在首次导出后持久化：

```python
def _get_embedding_function(self):
    if self._embedding_fn is None:
        from sentence_transformers import SentenceTransformer
        try:
            model = SentenceTransformer(self.embedding_model, backend="onnx")
            try:
                model.save_pretrained(self.embedding_model)   # 缓存 onnx，避免每次启动重导
            except Exception:
                pass
            log.info(f"[VectorStore] ONNX 加速已启用: {self.embedding_model}")
        except Exception as e:
            model = SentenceTransformer(self.embedding_model)
            log.warning(f"[VectorStore] ONNX 不可用，回退 PyTorch: {e}")
        self._embedding_fn = model.encode
    return self._embedding_fn
```
> `save_pretrained` 若 `embedding_model` 是 HF repo id 且在离线环境，可能写不进；用 `try/except` 包住即可，失败不影响主流程。保留 PyTorch 回退网。

### P1.4 验证

```bash
# 启用后首次启动日志应出现
# [VectorStore] ONNX 加速已启用: BAAI/bge-small-zh-v1.5
# 对比：上传同一文档，记录解析耗时（ONNX 应明显快于 PyTorch 回退）
```
- 加一个计时断言（解析 N 个 chunk 的耗时）纳入 `tests/` 或手动基准。

### P1.5 风险

| 风险 | 应对 |
|------|------|
| 离线环境无法导出 ONNX | 模型已本地缓存可本地导出；若仍失败自动回退 PyTorch，不阻断 |
| 依赖体积增大 | `onnxruntime` 约数十 MB，可接受 |
| 回滚 | 从 `requirements.txt` 删去 `[onnx]` 即回 PyTorch |

### P2.2 五级掌握度对齐（真正的增量）

我们 `progress_routes.py` 的 `calc_mastery_level`（文档级）与同学定义**几乎一致**（L2/L3/L4 阈值都是 60/80/90，次数 1/2/3），但同学有两处更严谨、我们可借鉴：

| 等级 | 同学定义 | 我们现状 | 差距 |
|------|---------|-----------|------|
| L2 熟悉 | 均分 60-79，**或** 测评≥2次且均分≥50 | 仅 `avg>=60` | 缺"≥2次且均分≥50"备选路径 |
| L3 精通 | **连续**2次≥80 **且薄弱点已消除** | 仅 `avg>=80` | 未要求"薄弱点消除"；用"平均"而非"连续" |
| L4 专家 | ≥3次均≥90 且所有知识点 mastery≥90% | `>=3次且>=90` | 一致 ✓ |

**修改点（在 `progress_routes.py` 的 `calc_mastery_level`）**：

```python
def calc_mastery_level(avg_score, assess_count, weak_count):
    # weak_count: 该文档下 L0/L1 知识点数量
    if assess_count == 0:
        return 0                                   # L0 未接触
    if assess_count >= 3 and avg_score >= 90:
        return 4                                   # L4 专家
    if assess_count >= 2 and avg_score >= 80 and weak_count == 0:
        return 3                                   # L3 精通（新增：薄弱点已消除）
    if assess_count >= 2 and avg_score >= 50:
        return 2                                   # L2 熟悉（新增：备选路径）
    if assess_count >= 1 and avg_score >= 60:
        return 2                                   # L2 熟悉
    if assess_count >= 1:
        return 1                                   # L1 入门
    return 0
```

> **"连续" vs "平均"的设计取舍**：同学用"连续 2 次≥80"（强调稳定性，需存每次测评分数历史）；我们用"平均≥80"（更鲁棒、无需历史）。建议**保留平均**，但可加一条"最近一次≥80"作为稳定性兜底，避免"一次 100、一次 40"拉平到 70 就判精通。若要做"连续"，需先确认 `assessments` 表是否记录了逐次分数（现有 `avg_score` 已够用于平均判定）。

**调用方改动**：`doc_mastery` 循环中计算 `weak_count`（该文档 L0/L1 知识点数），传入 `calc_mastery_level`。改动集中在 `progress_routes.py` 一处。

**知识点级阈值**（可选微调）：现状 `rate>=90→L4 / >=70→L3 / >=40→L2 / enc>0→L1`，与文档级不同轴（知识点用正确率、文档用测评均分），保持即可，无需对齐。
