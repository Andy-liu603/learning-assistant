## 2. 产品概述

### 2.1 产品定位

学习助手是一个**面向知识工作者的 RAG 增强全科学习系统**。它帮助用户将学习资料转化为结构化知识体系，通过 AI 驱动的问答、测评、进度追踪和学习计划生成，实现从"被动阅读"到"主动掌握"的学习闭环。

与通用 AI 聊天工具不同，学习助手的回答**严格基于用户上传的资料**，信息来源可追溯，避免幻觉；与传统的笔记工具不同，它不仅存储知识，更通过测评和掌握度追踪**量化学习效果**。

### 2.2 产品愿景

> 让每个人拥有 AI 驱动的个性化学习教练。

学习助手的长期愿景是成为一个**认知增强工具**——不是替代思考，而是放大人的学习能力。它做三件事：

1. **降低信息门槛**：复杂资料通过 RAG 对话变可理解
2. **建立反馈闭环**：测评揭示真实掌握程度，打破"看了就以为自己懂了"的幻觉
3. **量化学习过程**：用数据证明学习投入的有效性

### 2.3 产品演进路径

| 版本 | 时间 | 定位 | 关键里程碑 |
|------|------|------|-----------|
| **v1.0** | 2025.04 | MVP 验证 | 文档解析 + ChromaDB 向量化 + 基础 RAG 问答。验证"基于自己资料的 AI 问答"是否可行。 |
| **v2.3** | 2026.04 | 可用的 AI PM 工具 | JWT 多用户认证 + 异步文档处理 + 多模型架构 + SSE 流式输出 + 多模态图片理解 + 测评系统。从前端到后端完全自研，纯 JS SPA。 |
| **v2.4** | 2026.07 | 通用化学习助手 | 品牌升级为"全科学习助手" + 学习仪表盘 + AI 学习计划生成 + 五级掌握度量化评估 + AI 资讯追踪。从聚焦 AI PM 场景扩展到通用学习场景。 |
| **v3.0** | 规划中 | 多人协作平台 | 多人协作空间 + OAuth 登录 + 知识图谱可视化 + 移动端适配 + 学习社区。 |

当前 v2.4 版本是对 v2.3 的"通用化"升级——将原本面向 AI PM 的垂直定位扩展为面向所有知识工作者的全科学习平台。核心架构保持不变，新增了仪表盘、学习计划等用户感知层面的功能模块。

### 2.4 核心价值主张

学习助手的核心价值不是"让 AI 帮你读"，而是**让学习从不可量化变为可量化**。

传统学习流程：阅读 → 理解（自我感知）→ 遗忘 → 重新阅读。这个过程缺乏反馈闭环，学习者无法客观评估自己真正掌握了多少。

学习助手的流程：上传资料 → RAG 问答理解难点 → AI 出题测评检验掌握度 → 薄弱点识别 → 针对性强化 → 进度仪表盘追踪。每一步都产生数据，学习效果可量化。

## 3. 市场分析

### 3.1 坦诚声明

这是一个**个人项目**，当前唯一用户是开发者本人（北航研究生）。以下市场分析不是要论证该产品有商业化前景，而是论证**它解决的需求是真实存在的**，产品设计方向有对标依据。

### 3.2 个人知识管理工具市场格局

个人知识管理（PKM）是一个成熟且持续增长的市场。以下是关键产品的价值证明：

| 产品 | 估值/规模 | 核心能力 | 证明的需求 |
|------|----------|---------|-----------|
| **Notion AI** | 100亿美元（2024） | AI 驱动的笔记与知识库 | 用户愿意为"AI+知识管理"付费 |
| **Anki** | 全球数千万用户 | 间隔重复记忆卡片 | 用户有"记住所学内容"的强需求 |
| **Readwise** | 数十万付费用户 | 阅读高亮聚合与回顾 | 阅读后的"二次消化"是真实痛点 |
| **RemNote** | 百万级用户 | 笔记+记忆卡片一体化 | 知识输入与记忆巩固应在一个工具内完成 |
| **Quizlet** | 6000万+月活 | AI 出题 + 记忆游戏 | AI 辅助的自我测评是高频刚需 |

这些产品分别验证了知识管理的几个关键环节：存储（Notion）、记忆（Anki）、回顾（Readwise）、测评（Quizlet）。但尚无产品将这四个环节整合为一个流畅完整的学习闭环。

### 3.3 差异化定位

| 对比维度 | 学习助手 | Notion AI | Anki | ChatGPT/DeepSeek |
|---------|---------|-----------|------|-----------------|
| **知识来源** | 用户自有资料（RAG） | 笔记+全网 | 用户自制卡片 | 预训练模型 |
| **答案可追溯** | 是（引用原文片段） | 部分 | 否 | 否 |
| **AI 自动出题** | 是，基于文档内容 | 否 | 否（需手动制卡） | 可出题但不可溯源 |
| **掌握度追踪** | 是（五级 L0-L4） | 否 | 简单（正确率） | 否 |
| **学习计划生成** | 是（基于薄弱点） | 否 | 否 | 可生成但无数据支撑 |
| **测评反馈闭环** | 完整（出题→作答→判分→薄弱分析→强化建议） | 无 | 半闭环 | 无闭环 |
| **部署方式** | 本地部署，数据私有 | SaaS | 本地/云端 | SaaS |
| **多模态理解** | 支持图片分析 | 部分支持 | 不支持 | 部分模型支持 |

**核心差异**：学习助手是一个"学习闭环"工具，而非"知识存储"或"通用问答"工具。它的独特价值在于**测评→薄弱分析→精准强化**的反馈机制。

### 3.4 为什么是"真需求"

尽管当前仅单个用户使用，但需求可迁移验证：

1. **张同学（开发者本人）** 在备考 AI PM 时，通过上传 3 篇产品管理文献，用 AI 出题自测，发现了阅读时以为理解但实际未掌握的概念。这是传统阅读无法提供的反馈——这个需求是真实且被验证的。

2. **知识工作者的共性痛点**：在信息过载时代，任何一个需要持续学习的领域（编程、医学、法律、金融），都存在"资料多但吸收效率低"的结构性问题。学习助手提供的 RAG+测评方案是通用解法。

3. **AI 原生学习工具的窗口期**：大模型普及后，"用 AI 学习"成为显性需求，但大多数产品停留在"让 AI 替你总结"的阶段，缺乏"让 AI 检查你是否真懂"的进阶能力。学习助手填补了这个空白。

## 5. 用户痛点与场景

### 5.1 四大核心痛点

#### 痛点一：信息过载 — "资料很多，但不知道从哪里开始"

知识工作者面临的信息环境：网盘里数百个 PDF、收藏夹里成堆的"稍后再看"、微信里转发的"有时间读"。资料积累速度远超消化速度，导致**收藏即学会**的虚假满足感。

**现有方案的局限**：文件夹分类无法解决"优先级"问题，手动整理时间成本高，且无法感知内容质量。

**学习助手的解法**：用户上传资料后，系统自动解析并向量化。通过 RAG 问答，用户可以直接问"这篇文章的核心观点是什么"，无需通读全文即可建立整体认知，再决定是否精读。

#### 痛点二：碎片化学习 — "学了但没记住，缺乏系统化"

碎片时间学习（通勤、午休）的问题在于知识获取碎片化，无法形成体系。几个不连续的 20 分钟学习片段，产生的知识是孤立的。

**现有方案的局限**：笔记工具只是信息容器，不提供"连接"能力；间隔重复工具（Anki）需要手工制卡，费时费力。

**学习助手的解法**：测评系统基于文档内容自动生成题目，覆盖选择、判断、简答三种题型。提交后 AI 自动判分并给出错题解析。知识点掌握度（L0-L4 五级体系）追踪每个概念的掌握情况，薄弱点自动识别。

#### 痛点三：缺乏反馈闭环 — "不知道自己真正掌握了多少"

这是最隐蔽但最致命的痛点。"读完了"不等于"学会了"，但多数工具体系里，用户唯一的反馈来自自我感知——而人类的自我感知在学习效果上极其不可靠（达克效应）。

**现有方案的局限**：考试是反馈，但不可高频使用；练习题耗时间且不够个性化。

**学习助手的解法**：测评→判分→薄弱分析→强化建议形成完整闭环。用户可以在读完一篇资料后立即生成测评，10 分钟内完成自测，获得客观的掌握度数据。这种**即时反馈**是传统学习方式无法提供的。

#### 痛点四：无法量化学习效果 — "投入产出比不透明"

学习是一项高时间成本活动，但效果难以量化。用户可能花了 20 小时学一个主题，却无法说清"我到底学成了什么"。这降低了学习的正反馈，削弱坚持动力。

**现有方案的局限**：番茄钟类工具只计时不计效；学习打卡只记录"是否学了"不记录"学会了什么"。

**学习助手的解法**：仪表盘提供多维度的学习数据可视化——90 天学习热力图、掌握度环形图、学习时长趋势折线图、薄弱知识点列表。周报系统自动汇总每周学习成果，提供数据驱动的时间投入建议。

### 5.2 典型用户场景

#### 场景 1：研究生研读论文

1. 上传导师指定的 3 篇英文论文
2. 系统自动解析（PDF 文本提取 + 图表通过视觉模型理解）
3. 对每篇论文发起 RAG 对话，提问核心概念和实验方法
4. 生成综合测评，验证阅读效果
5. 导出周报中的论文阅读总结

#### 场景 2：跨领域求职准备

1. 收集目标领域的入门资料（电子书、行业报告、课程课件）
2. 全部上传到学习助手
3. 按主题建立对话（如"产品管理基础知识""大模型技术原理"）
4. 每学完一个模块做测评
5. 根据薄弱点生成个性化学习计划
6. 仪表盘追踪整体学习进度

#### 场景 3：期末集中复习

1. 按课程分类上传整学期课件
2. 每门课生成 20 道 AI 测评题
3. 完成测评后查看薄弱知识点
4. 对每个薄弱点生成专项练习题
5. 反复练习直到掌握度达到 L3（精通）以上
6. 复习周报记录复习效果

## 7. 非功能需求

### 7.1 性能要求

| 指标 | 目标值 | 测量方法 | 当前状态 |
|------|--------|---------|---------|
| RAG 检索延迟 | < 2 秒 | 向量搜索 + 上下文组装耗时 | 本地 ChromaDB + BGE 嵌入，实测 < 1s |
| LLM 首 token 延迟 | < 3 秒 | 从发送请求到收到首个 SSE chunk 的时间 | 取决于 API 提供商，DeepSeek 实测 < 2s |
| 文档处理吞吐 | < 30 秒/标准 PDF（10 页） | 上传完成到 `status: ready` 的时间 | 异步处理，后台完成 |
| 前端首屏加载 | < 2 秒 | Lighthouse Performance 评分 | 纯 JS SPA，零外部依赖，接近即时 |
| API 响应时间 | < 200ms（非 LLM 接口） | SQLite 查询 + 序列化 | SQLite + WAL 模式，实测 < 50ms |

### 7.2 安全要求

| 要求 | 实现方式 | 说明 |
|------|---------|------|
| 身份认证 | JWT (HS256)，7 天过期 | 所有非公开 API 需 Bearer token |
| 用户数据隔离 | SQLite 全表 `user_id` 字段 + ChromaDB 按用户分 collection | 杜绝跨用户数据泄露 |
| 密码安全 | bcrypt 哈希存储 | 不存储明文密码 |
| API 速率限制 | 暂未实施（目标 v3.0 补充） | 当前为单用户本地部署，风险可控 |
| 敏感信息保护 | API Key 通过环境变量注入，不入代码库 | `.env` 文件已在 `.gitignore` 中 |
| CORS | 限制为 localhost 来源 | 本地部署场景足够 |

### 7.3 可用性要求

| 要求 | 实现方式 |
|------|---------|
| 零配置本地部署 | 免安装数据库（SQLite + ChromaDB 嵌入式），解压即用 |
| 一键启动 | `pip install -r requirements.txt && python backend/app.py`，Windows 提供 `start.bat` |
| 环境隔离 | 使用 Python 虚拟环境（`.venv`），避免系统环境冲突 |
| 前端兼容性 | 纯 JS SPA，不依赖任何前端框架，现代浏览器即可运行 |

### 7.4 可维护性要求

| 要求 | 实现方式 |
|------|---------|
| 代码结构 | 后端：Flask Blueprint 模块化路由 + Service 层分离；前端：ES Modules + Hash 路由 |
| 日志系统 | 独立日志模块，按日期轮转，支持分级（DEBUG/INFO/WARNING/ERROR） |
| 数据库迁移 | SQL 迁移脚本版本化存储在 `backend/migrations/` |
| 配置管理 | 集中式 `config.py` + `.env` 环境变量，支持多模型 Provider JSON 配置 |
| 项目文档 | PROJECT.md（完整架构文档）、README.md（快速入门）、CLAUDE.md（AI 协作规范） |

## 9. 成功指标

### 9.1 说明

作为个人项目，传统产品的成功指标（DAU、留存率、付费转化）不适用。以下指标分为三类：系统质量、使用行为和效果验证，均可在单用户场景下收集和评估。

### 9.2 系统质量指标

| 指标 | 测量方法 | 目标值 | 当前状态 |
|------|---------|--------|---------|
| RAGAS 忠实度（Faithfulness） | 人工标注 50 个问答对，评估回答是否忠于原文 | ≥ 0.85 | 待评估 |
| RAGAS 答案相关性（Answer Relevancy） | 同上，评估回答与问题的匹配程度 | ≥ 0.80 | 待评估 |
| RAGAS 上下文精度（Context Precision） | 同上，评估检索结果中相关片段的比例 | ≥ 0.75 | 待评估 |
| API 可用率（健康检查） | 24 小时持续运行监控 | ≥ 99% | — |
| 无重大 Bug 天数 | 记录影响核心功能（上传/问答/测评）的 Bug 修复周期 | ≤ 7 天修复 | — |

### 9.3 使用行为指标

| 指标 | 定义 | 数据来源 |
|------|------|---------|
| 周学习天数 | 至少 1 次有效学习活动的天数/周 | `study_sessions` 表 |
| 月文档上传量 | 新增处理的文档数量/月 | `documents` 表 |
| 测评完成率 | 已完成测评数 / 创建测评总数 | `assessments` 表 |

### 9.4 学习效果指标（自实验对比）

这是最具说服力的指标，通过开发者本人的自我实验获取：

## 附录 A：术语表

| 术语 | 全称 | 定义 |
|------|------|------|
| **RAG** | Retrieval-Augmented Generation | 检索增强生成：从文档库中检索相关内容，注入 LLM 上下文后生成回答 |
| **SSE** | Server-Sent Events | 服务器推送事件：单向流式数据传输，用于 LLM 逐字输出 |
| **JWT** | JSON Web Token | 基于 JSON 的认证令牌，用于无状态用户身份验证 |
| **ChromaDB** | — | 开源向量数据库，基于 HNSW 索引实现高效语义搜索 |
| **Embedding** | — | 文本嵌入向量：将文本映射到高维向量空间，用于语义相似度计算 |
| **L0-L4 掌握度** | — | 五级知识掌握度评定体系：未接触/入门/熟悉/精通/专家 |
| **RAGAS** | RAG Assessment | RAG 系统评估框架，包含忠实度、相关性、精度等维度 |
| **SPA** | Single Page Application | 单页面应用：无需整页刷新，前端动态渲染 |
| **HNSW** | Hierarchical Navigable Small World | 分层可导航小世界图，ChromaDB 默认的近似最近邻搜索算法 |
